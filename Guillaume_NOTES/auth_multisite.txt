---

# 🔥 Pourquoi ajouter `onsite` est une excellente idée

Dans un contexte d’entreprise réelle :

- une entreprise peut avoir **plusieurs sites physiques**  
- chaque site peut avoir **ses propres workers**  
- chaque site peut avoir **ses propres jobs**  
- chaque site peut avoir **ses propres permissions**  
- certains workers peuvent être **multi‑site**, d’autres non  

Donc oui, ajouter un champ `onsite` (ou même un objet complet) est **parfaitement cohérent**.

---

# 🧱 Version simple (ta demande exacte)

Tu peux faire :

```json
{
  "type": "auth.login",
  "enterprise": {
    "id": "123",
    "name": "garage123",
    "onsite": "123 Rue Industrielle, Montréal, QC"
  },
  "identity": {
    "email": "worker@garage123.ca",
    "username": "worker42"
  },
  "password": "secret123"
}
```

Ça marche très bien.

---

# 🧱 Version évoluée (multi‑site professionnel)

Si tu veux un système **multi‑succursales**, tu peux aller vers :

```json
{
  "type": "auth.login",
  "enterprise": {
    "id": "123",
    "name": "garage123",
    "sites": [
      {
        "id": "S1",
        "name": "Garage 123 - Montréal",
        "address": "123 Rue Industrielle, Montréal, QC",
        "timezone": "America/Toronto"
      },
      {
        "id": "S2",
        "name": "Garage 123 - Laval",
        "address": "55 Boulevard Industriel, Laval, QC",
        "timezone": "America/Toronto"
      }
    ]
  },
  "identity": {
    "email": "worker@garage123.ca",
    "username": "worker42",
    "siteID": "S1"
  },
  "password": "secret123"
}
```

### Pourquoi c’est puissant ?

- tu peux gérer **plusieurs sites** par entreprise  
- tu peux associer un worker à **un site spécifique**  
- tu peux gérer des permissions par site  
- tu peux faire du **job routing** par site  
- tu peux faire du **load balancing** par site  
- tu peux faire du **monitoring** par site  

C’est exactement ce que font les systèmes professionnels (AWS Organizations, Azure Tenants, Google Workspace, etc.).
